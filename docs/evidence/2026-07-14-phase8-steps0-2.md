### Design decisions (recorded)

- **QR/PDF library versions corrected during implementation.** The 2026-07-13
  research note (`PLAN.md`, ADR 0007) cited ZXing 3.5.3 from a web search;
  Maven Central metadata checked at implementation time showed 3.5.4 already
  released. Pinned to the actual current release rather than the stale
  research figure.
- **The label payload is not re-derived.** `LabelService` reads the existing
  `location.qrValue` / `article.qrValue` columns (already `LOC:<code>` /
  `ART:<sku>` per the Phase 7 schema) rather than reformatting the code/SKU
  itself, so the label generator and the HHT scan endpoints are provably
  reading the same value — the FT-17 round-trip test exploits exactly this.
- **PDF determinism required inspecting the actual PDFBox 3.0.8 source**, not
  assuming `sendError`-adjacent framework behavior. `COSWriter` seeds its
  trailer-ID SHA-256 hash with `PDDocument.getDocumentId()` (falls back to
  wall-clock time if unset) plus the `/Info` dictionary's values; both are now
  fixed (`document.setDocumentId(0L)`, a constant `PDDocumentInformation`),
  verified by the FT-17 byte-equality assertion rather than assumed.
- **The dashboard is a second, independent `SecurityFilterChain`**, not a
  branch inside the existing bearer chain — a session/cookie/CSRF-protected
  surface and a stateless bearer-token surface have materially different
  session, CSRF, and error-handling needs, matching the separation the ADR 0007
  consequence already anticipated ("the browser dashboard's cookie/CSRF story
  is a later phase").
- **Access-denied handling writes the response directly** instead of the
  Spring Security default `response.sendError(403)` — see Deviations below.

`LabelApiIT` proves, end to end: PNG and PDF generation for both a location
and an article is byte-identical across two independent calls; the PNG's QR
payload, decoded with ZXing's own reader, equals the exact `LOC:<code>` /
`ART:<sku>` string; that decoded value is then submitted to the *real*
`POST /hht/tasks/{id}/scan-location` and `.../scan-article` endpoints for a
freshly claimed task and accepted (`200`, correct state transition); a
`PICKER` gets `403` on the label endpoints; an unknown code/SKU gets the
existing `404 LOCATION_NOT_FOUND` / `404 ARTICLE_NOT_FOUND` problem, not a raw
error.

`DashboardApiIT` proves: an unauthenticated `GET /dashboard` redirects to
`/login` rather than serving content; wrong credentials redirect back to
`/login`, not `/dashboard`; a valid `ADMIN` session sees a `200` page whose
server-rendered HTML contains a freshly seeded order, and `GET
/dashboard/api/tasks` returns the same live state as JSON; an authenticated
`PICKER` session is refused both the page and the poll endpoint with `403`.

## Deviations and notes

- **A real cross-chain bug was found and fixed during this step, not a test
  artifact.** The dashboard's access-denied handling initially relied on
  Spring Security's default `AccessDeniedHandlerImpl`, which calls
  `response.sendError(403)`. That triggers a servlet-container forward to
  `/error` — a *new* request URI that no longer matches the dashboard chain's
  `/dashboard/**` `securityMatcher` and falls through to the stateless bearer
  chain instead. That chain sees no `Authorization` header on the forwarded
  request, treats it as unauthenticated, and its own `WmsAuthenticationEntryPoint`
  answers `401 application/problem+json` — silently turning every dashboard
  `403` into a `401` with a JSON body on an HTML page. Caught by
  `DashboardApiIT`'s explicit `assertEquals(403, …)` check, not by
  compilation or a happy-path run. Fixed by giving the dashboard chain its
  own `accessDeniedHandler` that sets the status directly
  (`response.setStatus(403)`), never invoking the container's error-dispatch
  mechanism. Recorded per the failed-experiment rule.
- **The same class of bug, found by the manual browser check, not the
  automated test.** Spring's default login page references
  `/default-ui.css` for styling; that path also isn't matched by the
  dashboard `securityMatcher`, so it 401-ed with a JSON body too (visible as
  a browser console MIME-type error and an unstyled login page), even though
  `DashboardApiIT` — which parses HTML with a regex rather than rendering it
  — had no way to notice. Fixed the same way, by adding `/default-ui.css` to
  both the `securityMatcher` and the `permitAll()` list. This is the
  concrete justification for doing the manual browser pass at all: an
  HTTP-assertion test proved the *security* behavior correct while missing a
  real, user-visible defect the acceptance gate's own "browser evidence"
  expectation was designed to catch.
- No other deviations from the established patterns (Jackson 3, `RestTemplate`
  + JDK `HttpClient` factory, per-class Testcontainers database).

- Step 3 (evidence/gate checkoff) closes with this record; `PLAN.md`'s Phase 8
  acceptance-gate boxes and `docs/requirements-traceability.md` are updated
  alongside it.
- The dashboard has no logout affordance (out of scope: no FT case exercises

  `stuckOnly`); acceptable for the "minimal" dashboard deliverable, revisit if
  the portfolio review wants filter controls in the page itself.

  into the npm/OS package cache for this session only; it is not a project
  dependency and nothing under version control depends on it.
